Design patterns


Description

Un design pattern est une solution standardisée, a un problème récurrent qu'on cherche à résoudre. Un design pattern est donc fait pour être réutilisé. C'est une approche éprouvée qui encapsule des principes, logique d'architectures et d'algorithmes pour résoudre un problème
Un data pipeline est un processus automatisé pour ingérer les flux de données, déplacer la donnée, la transformer, la stocker et éventuellement la supprimer pour terminer son cycle de vie

Traitement en batch


C'est le traitement de données en lots, le but est de prendre des données en grande quantités d'un point A à un point B, ça permet de traiter des grands volumes de données en incrémental et avec toute logique d'Idempotence.
Le processing et le traitement se fait toutes les heures ou toutes les 12h, 24h c'est généralement dans ces ordres de pattern qu'on va appliquer le traitement en Batch.

Le problème du traitement en Batch, c'est qu'il est plutôt lent, car il s'agit de traiter de très gros volumes de données d'un coup. Ce n'est donc pas adapté si on veut transformer les données de manière rapide.

C'est un pattern qui va être utilisé pour traiter des gros volumes de données mais en petits morceaux, on rassemble les données en micro batchs, de manière scalable, et pour réduire le temps de traitement au lieu d'avoir un batch toutes les heures ou 24h ça sera par exemple toutes les minutes.

Kappa Architecture



La Kappa Architecture est un modèle d'architecture de traitement des flux de données en temps réel. Elle a été proposée par Jay Kreps, l'un des créateurs d'Apache Kafka.
Le principal principe de la Kappa Architecture est d'utiliser un système de streaming pour traiter les données en temps réel de manière continue, sans distinction entre les flux entrants et les données historiques. Cela permet de simplifier l'architecture en éliminant la nécessité d'un système de traitement distinct pour les données en temps réel et les données historiques. Les données sont ingérées dans un flux continu, puis elles sont traitées de manière uniforme à mesure qu'elles arrivent. Il y a 3 couches l'architecture Kappa toutes en streaming temps réel, une première pour Ingérer, une deuxième pour Transformer et une troisième couche de service

Lambda Architecture



La Lambda Architecture est un modèle d'architecture de traitement des données qui combine le traitement en stream temps réel et le traitement en batch temps différé pour fournir une vue complète des données.
Elle a été proposée par Nathan Marz. La Lambda Architecture repose sur la notion de deux voies parallèles de traitement des données : une voie en temps réel et une voie en temps différé. Les données sont ingérées simultanément dans les deux voies. La voie en temps réel est utilisée pour le traitement en continu des données en temps réel, tandis que la voie en temps différé est utilisée pour le traitement batch des données historiques. Les résultats des deux voies sont ensuite fusionnés pour obtenir une vue complète des données c'est le croisement entre le traitement en batch et le traitement en stream.
Pour l'architecture Lambda, 3 couches sont utilisées : une première c'est la couche batch la ou on va recupérer les données, les transformer en batch et les stocker, la 2ème couche est en stream, on va récupérer les données en temps réel, les transformer et les afficher en temps réel. la 3eme couche qui est la couche de présentation ou de service qui permet de fusionner les 2 approches, données en streaming et en batch, donc finalement cette couche de présentation va permettre de consolider la partie batch et la partie streaming pour avoir des résultats rapides avec la couche streaming et précis avec la couche batch.

Data Engineer vs Data Scientist



Data Scientist

Un Data Scientist utilisera en général les résultats principaux d'algèbre linéaire et multivariable pour remodeler et créer des nouveaux aperçus des données.
À quoi ressemblent ces nouveaux aperçus ?

Une entreprise fabrique des produits qui doivent être testés avant expédition.
Généralement, ces tests prennent beaucoup de temps car il y a des centaines de choses à tester.
Tout cela pour vous assurer que votre produit n'est pas cassé.
Ça serait intéressant de savoir très tôt, avec le moins de tests possibles, si le produit n'est pas conforme et qu'il faut le réparer ou simplement le jeter, afin de gagner en temps et en ressources pour faire tous le reste des tests.
C’est exactement là qu’un data scientist peut vous aider grandement. Ce champ est appelé analyse prédictive, et la technique pour choisir les tests nécessaires c’est l’apprentissage automatique, Machine learning.
Machine quoi ? Learning?
Oui, le machine learning, ça marche comme ça :
Vous alimentez un algorithme avec des données pertinemment préparées. Il génère un modèle et l'optimise en fonction des données avec lesquelles vous l'avez alimenté. Ce modèle montre essentiellement comment vos données sont distribuées, c’est à dire à quoi elles ressemblent, vous montrez à ce modèle de nouvelles données et le modèle vous dira si ces nouvelles données suivent ce modèle. Cette technique peut également être utilisée pour prédire à l’avance les pannes des machines grâce à l’apprentissage automatique. Bien sûr, tout le processus n'est pas si simple.
Le plus grand travail du Data Scientist consiste à comprendre comment prétraiter les données avant de les injecter à l'algorithme.
De l'autre coté, les sorties du modèle sont très abstraites, et il faut un post-traitement des données de sortie.


Data Engineer

Les Data Engineers assurent les liens entre les stratégies et les choix Big Data adoptées et les Data Scientists et leurs besoins, ils créent les plateformes qui permettent aux Data Scientists de construire les modèles les plus pertinents selon le cas d’usage
Ces plateformes sont généralement utilisées de quatre manières différentes :

  • Ingestion et stockage de larges volumes de données
  • Création d'algorithmes de machine learning
  • Automatisation des modèles et algorithmes d’apprentissage automatique pour une utilisation en production
  • Data viz, tableaux de bords et reporting pour les employés, les utilisateurs ou le client
La plupart du temps, ces personnes commencent en tant qu'architectes de solutions traditionnelles pour les systèmes qui impliquent des bases de données SQL, des serveurs Web, des installations SAP et d'autres systèmes « standards »
Mais pour créer des plateformes Big Data, l'ingénieur doit être un expert dans la spécification, la configuration et la maintenance des technologies Big Data telles que : Hadoop, Spark, HBase, Cassandra, MongoDB, Kafka, Redis et plus encore.
Ce dont ils ont également besoin, c'est d'une expérience sur la façon de déployer des systèmes sur une infrastructure cloud comme sur Amazon ou Google ou sur du matériel spécifique.
